# Diamonds Predictor Application

An end-to-end machine learning project on the Kaggle Diamonds dataset (~54,000 stones):
exploratory analysis, feature engineering, three models, and a natural-language assistant
that answers questions about the data.

## Methodology

### Data cleaning
The dataset ships with three defects that `isna()` will not catch:

- **~146 exact duplicate rows.** They allow the same stone to land in both train and test,
  quietly inflating every reported score.
- **~20 rows with `x`, `y`, or `z` equal to 0.** A zero-millimetre diamond does not exist.
  This is a missing value disguised as a number вЂ” the reason you check numeric ranges against
  physical reality, not just for nulls.
- **A handful of absurd dimensions** (`y` = 58.9 mm, `z` = 31.8 mm) on ordinarily-priced rows.
  Data-entry errors, and enough on their own to tilt a regression line.

### Encoding вЂ” ordinal, deliberately
`cut`, `color`, and `clarity` are **quality grades with a real ranking** (Ideal > Premium >
Very Good > Good > Fair). Ordinal encoding *records* that order; one-hot encoding would
*destroy* it and force the network to rediscover it from disconnected binary columns.

This is the opposite of the correct choice for an unordered category like `region` or `sex`,
where any integer mapping invents a false ranking. **The rule: one-hot when categories are
unordered, ordinal when the order is real.** Look at the data; don't apply a reflex.

### Feature engineering
- `volume` = xВ·yВ·z вЂ” carat is a *weight*, volume is a *size*; a poor cut is bulky for its weight
- `density` = carat / volume вЂ” a deep, weight-retaining cut
- `xy_ratio` = x / y вЂ” symmetry deviation, a measurable cut defect no raw column captures
- `log_carat` вЂ” straightens the multiplicative caratв†’price relationship

### Leakage control
Each task gets its **own feature set**:

- **Classification** uses `price`, excludes `clarity_rank`
- **Regression** uses `clarity_rank`, excludes `price`, `log_price`, and `price_per_carat`
- **Clustering** uses no target at all

`price_per_carat` deserves a specific warning: it is a real gemmological quantity, and feeding
it to the price model produces **RВІ = 1.00** вЂ” because `price_per_carat Г— carat` *is* `price`,
exactly. The model has learned to multiply, not to value diamonds, and at prediction time you
would not have it. **The test for any feature: would I actually have this value at the moment
I need to make the prediction?**

### Modelling
- **Classification** вЂ” 128в†’64в†’32в†’8, ReLU, BatchNorm, Dropout 0.3, **softmax** output,
  `sparse_categorical_crossentropy`, **balanced class weights** (clarity is badly imbalanced;
  I1 is under 1.5% of rows)
- **Regression** вЂ” 256в†’128в†’64в†’32в†’1, **linear** output, MSE loss, trained on **log(price)**
- Both use **EarlyStopping** with `restore_best_weights=True` and **ReduceLROnPlateau**, so the
  epoch count is decided by the validation set rather than guessed
- **Three-way split** (70/10/20). The validation set is what EarlyStopping watches вЂ” if it
  watched the test set, the test set would no longer be held out
- Scalers are fit on **train only** and shipped with the models

### Why log(price)
Price is heavily right-skewed. A network minimising MSE on raw price is minimising *squared
dollars*: a $2,000 miss on an $18,000 stone contributes 100Г— more loss than a $200 miss on a
$500 stone. But $200 out on a $500 diamond is a catastrophic 40% error, and $2,000 out on an
$18,000 one is a tolerable 11%. **Raw-dollar MSE has its priorities backwards.** Training on
log(price) makes a *proportional* error cost the same at every scale вЂ” so the model optimises
percentage accuracy, which is what a jeweller cares about. Metrics are reported back in dollars.

Evidence it worked: percentage error stays roughly flat across price bands (notebook 04).

## The AI assistant

Three stages, with the LLM confined to the edges:

| Stage | Who | Touches the data? |
|---|---|---|
| **Understand** | LLM в†’ structured JSON filter | No |
| **Retrieve** | Our pandas code, whitelist-validated | Yes |
| **Explain** | LLM narrates *our* computed facts | No |

The naive approach вЂ” have an LLM write pandas code and `exec()` it вЂ” is arbitrary code execution
pointed at your own data. This design makes that impossible: every column and operator the model
emits is checked against a whitelist before anything reaches the dataframe.

It also kills hallucination almost for free. In the explain stage the model is handed the numbers
*we* computed and asked to narrate them. It is never asked to produce a figure, so it has no
opening to invent one.

**Treat the LLM as an untrusted component inside your own system**, not as a trusted colleague.

Falls back to a keyword parser and a template when no API key is present.

### The finding that matters

Same architecture, same effort, same data вЂ” **wildly different outcomes**.

The price regressor reaches RВІ в‰€ 0.98. The clarity classifier roughly doubles its baseline and
stalls. Neither result is about the model.

**Price is physically determined by the features we have** вЂ” carat alone correlates above 0.9
with it. **Clarity is not.** Clarity is a judgment about microscopic inclusions *inside* the
crystal, and nothing in carat, depth, table, x, y, z, cut or colour measures them. We are
inferring an optical property from geometry and cost.

> A model's quality is bounded by the information in its features.
> No amount of architecture search conjures signal that isn't in the data.

When a model underperforms, the first question is never *"which optimiser?"* вЂ” it is
**"is the answer even in the inputs?"**

## Known limitations

- **Clarity model** вЂ” the ceiling is structural, not a tuning problem. It uses `price` as an
  input, which is fine for sanity-checking a *listed* stone but makes it unsuitable for grading
  an unpriced one.
- **Price model** вЂ” one snapshot of one market. No brand, no certification body, no fluorescence
  or polish grade, no price index. Extrapolates badly outside the carat range it saw.
  **A gradient-boosted tree would likely match or beat it** on tabular data with far less tuning;
  the ANN is the assignment's requirement, not the objectively best tool for this problem shape.
- **Segmentation** вЂ” the diamonds data is a *continuous cloud*, not naturally separated blobs.
  K-Means always returns k clusters whether or not k exist, so these tiers are **imposed**, not
  **discovered**. Useful as a business abstraction; not a law of nature. (DBSCAN correctly fails
  on this data for exactly that reason вЂ” see notebook 05.)
- **"Customer" segmentation** вЂ” the dataset contains no customers, only diamonds. What is
  actually delivered is *inventory* segmentation into product tiers, which is the thing a
  merchandiser would build price bands around.

## Next steps

**Features first** (highest expected return): inclusion counts, fluorescence, polish and symmetry
grades, certification body. The clarity ceiling is a data problem and only better data will move it.

**Then models**: benchmark against XGBoost / LightGBM; Keras Tuner or Optuna instead of
hand-picked hyperparameters; 5-fold CV so scores come with error bars.

**Then evaluation**: cost-sensitive metrics (misgrading an I1 as an IF is commercially serious;
VS1 as VS2 is not вЂ” plain accuracy treats them identically). Prediction *intervals* on price вЂ”
"$4,200 В± $600" is more honest and more usable to an appraiser than "$4,200".
